# БЛОК 01 — Core Engine (Главное ядро)

**Назначение:** Главный диспетчер конвейера. Принимает запрос от флагмана, управляет
всеми этапами: поиск символов → сборка картриджа → выдача лучшего решения → верификация
→ архивация. Работает на отдельном сервере. Флагман работает на стороне клиента.

## Принцип работы


### AI Router — единственная точка вызова моделей

Все процессы ядра требующие участия ИИ идут через единый коннектор.
Конфигурация: `config/ai_router.json`

```python
# Все AI-вызовы внутри ядра — через этот коннектор
from core.ai_router import ai_router

# Векторизация — делает локальная модель ЯДРА (не флагман)
vector = await ai_router.embed(text="label + science + section + subsection")

# Классификация Тип А/Б
result = await ai_router.classify(prompt=obsidian_prompt, task="type_ab")

# Верификация YMS-MMM
verdict = await ai_router.verify(prompt=yms_prompt, task="verification")
```

Векторизацию выполняет **локальная модель ядра на стороне сервера**.
Флагман не участвует в векторизации — он на стороне клиента.
При сбое основной модели — автоматический fallback по цепочке из `ai_router.json`.

**Ядро снабжает знаниями — не контролирует флагмана.**
Флагман получает лучшее проверенное решение (наивысший R_f) и сам решает как применять.

**Безопасность ответов:** каждый HTTP-ответ ядра подписан HMAC-ключом сервера.
Флагман верифицирует подпись на входе автоматически.
Пользователь видит все ответы и взаимодействие.

**Запись — миллисекунды.** Асинхронная очередь Redis Queue.
zstd сворачивается на лету в памяти — без промежуточных файлов на диск.
Пользователь не ждёт записи никогда.

## Конвейер Фаза 0 — Минимальный эндпоинт

```
POST /api/v1/query
{
  "user_request": "...",
  "flagship_plan": ["шаг 1", "шаг 2", ...],
  "session_id": "..."
}
```

1. Векторизовать `user_request` — силами подключённой к ядру модели
2. Поиск в БЛОК 02: top-5 символов (similarity × log(R_f+2))
3. Выдать флагману символ с наивысшим R_f + описания всего набора по метаданным
4. При выполнении каждого шага: декомпрессия zstd из БЛОК 03 → выдача инструкции
5. Ответы подписывать HMAC

**При сбое векторизации:**
```
1. Зафиксировать: «В системе что-то изменилось»
2. Перейти в режим поиска нового решения (как при cartridge_empty)
3. Найти актуальный метод во внешних источниках
4. Верифицировать через YMS-MMM → записать → применить
```

## Конвейер Фаза 1 — Полный

### Запрос плана
```
POST /api/v1/query (step 1, без плана) → { "status": "plan_required" }
POST /api/v1/query (step 2, с планом)  → поиск + картридж
```

### Консьерж + Query: один маршрут вместо двух

Стек определяется через консьерж **до** `/api/v1/query`.
Никакого отдельного `stack_query` — всё приходит в одном запросе:

```
POST /api/v1/concierge   ← 1й round-trip: выяснить стек и контекст
POST /api/v1/query       ← 2й round-trip: план + стек → ювелирный поиск
```

Итого 2 запроса вместо 3-4. Токены флагмана тратятся один раз.
Локальная модель получает полный контекст сразу и ищет точно.

### Эндпоинт step_done: последовательное раскрытие шагов

```
POST /api/v1/step_done
{
  "session_id": "<ID>",
  "step_completed": 1,
  "step_result": "success",
  "next_step_requested": 2
}
```

Ядро получает сигнал что шаг N завершён → декомпрессирует zstd следующего символа
→ выдаёт флагману детальную инструкцию шага N+1.
Следующий шаг раскрывается **только** после подтверждения предыдущего.

### Три сценария картриджа

**Сценарий А — Комплексная лигатура есть:**
- Выдать лигатуру с наивысшим R_f
- Фрактально найти составные символы
- Загрузить весь набор в pgvector/RAM
- Раскрывать по очереди по мере выполнения шагов

**Сценарий Б — Модульная сборка:**
- Найти символы под каждый шаг, собрать лигатуру на месте
- Записать в библиотеку асинхронно
- Раскладывать на составные, выполнять по очереди
- Проверять эффективность после каждого шага

**Сценарий В — Белое пятно:**
- Выдать директиву флагману: найти во внешних источниках
  (GitHub высокорейтинговые → официальная документация → маркетплейсы)
- После верификации — записать в библиотеку

### Последовательное раскрытие
```
Весь набор символов загружается в pgvector/RAM СРАЗУ при инициализации.
Горячие (повторяющиеся в плане) → Redis.
Детальная декомпрессия zstd каждого символа — только когда предыдущий шаг выполнен.
Позиция символа в плане — динамическая, определяется из соответствия шагам.
Метаданные символа — статичны, описывают суть как пункт оглавления.
```

### Верификация YMS-MMM
```
Флагман отчитался → БЛОК 06 проверяет:
  PASS → контур Obsidian → архивация (async) → хук-допрос
  FAIL × 1-2 → fix_directive → флагман исправляет
  FAIL × 3 → webhook → БЛОК 07 (Gemini → Ollama fallback) → патч
```

### Контур Obsidian (архивация)
```
Тип А (улучшение):
  - Новое тело → zstd на лету → БЛОК 03 (async queue)
  - Старый символ ID → в legacy_symbols[] основного
  - R_f += 1
  - Переходный маяк: «было [X] → стало [Y]»

Тип Б (новые обстоятельства):
  - Новый символ → evolved_from → БЛОК 02 create_symbol()
  - applicable_stacks[] заполнить по контексту задачи
```

### Схлопывание сессии
```
Триггер: тема исчерпана / сессия закрыта
1. Async запись если есть новое знание
2. Редактирование активных символов (если найдено лучшее)
3. Схлопывание в обратном порядке активации:
   Слой 1 (активные шаги) → Слой 2 (план) → Redis горячий
4. Физическое удаление из RAM. Данные — только на шардах.
5. Изоляция: разные пользователи не пересекаются в памяти.
```

## HMAC-подпись ответов

```python
import hmac, hashlib, json

def sign_response(payload: dict, secret_key: str) -> dict:
    body = json.dumps(payload, sort_keys=True)
    signature = hmac.new(
        secret_key.encode(),
        body.encode(),
        hashlib.sha256
    ).hexdigest()
    return {**payload, "evo_signature": signature}

def verify_response(payload: dict, secret_key: str) -> bool:
    sig = payload.pop("evo_signature", None)
    if not sig:
        return False
    body = json.dumps(payload, sort_keys=True)
    expected = hmac.new(
        secret_key.encode(),
        body.encode(),
        hashlib.sha256
    ).hexdigest()
    return hmac.compare_digest(sig, expected)
```

Флагман вызывает `verify_response()` при получении каждого HTTP-ответа.
Подпись — внутренний протокол сервер↔флагман. Пользователь видит чистые ответы.
