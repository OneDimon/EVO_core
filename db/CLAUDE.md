# db/ — PostgreSQL + pgvector layer (Блок 02)

**Перед правкой обязательно прочитать:** `../BLOCK_02_language_library.md`,
а также сами SQL-файлы миграций по порядку (001 → 008) — не полагаться на
память о схеме, только на факт.

## Что здесь
- `migrations/001_init.sql` … `008_fix_ivfflat_cleanup.sql` — применённые
  миграции. **Следующий свободный номер — 009.**
- `models.py` — Python-модели/типы, соответствующие таблицам.
- `pg_client.py` — asyncpg-пул, `find_symbols`, `get_symbol`, `increment_rating`.
- `redis_client.py` — кэш символов, rate-limit счётчики.
- `users.py` — пользователи/сессии.

## Правила
- **Никогда не редактировать уже применённые миграции 001-008** — только
  новая миграция для изменений схемы.
- Индекс pgvector — HNSW (см. migration 006/008 — там была история с двумя
  дублирующимися индексами `ivfflat`/`hnsw`, сверяй реальное имя индекса
  через `\d` в psql перед DROP/CREATE, не по памяти).
- Любое новое поле, которое хранит секрет/токен — регистрировать в
  `core/crypto.py::SENSITIVE_KEYS`, писать в БД только через `config_manager.py`.
- Docker-compose монтирует миграции в `docker-entrypoint-initdb.d` — они
  применяются автоматически **только при первом создании volume**; для
  существующей БД накатывать вручную (см. корневой `CLAUDE.md`).
