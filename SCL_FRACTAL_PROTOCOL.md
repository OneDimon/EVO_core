# SCL Fractal Protocol v2.1 — Полный протокол Языка-Библиотеки EVO-core

> **Статус:** Фундаментальный документ. Защищён YMS-MMM.
> Не изменяется без явного разрешения Архитектора.

---

## 1. Что такое Язык-Библиотека

Язык-Библиотека — когнитивное ядро EVO-core. Собственный язык системы из лигатур, отпечатков, строчных/надстрочных символов и их фиксированных цифровых значений. Символы **сами по себе являются векторами**. В метаданных каждого символа зашиты микро-маячки — они позволяют локальной модели резонировать с сутью запроса мгновенно, без тяжёлого семантического поиска.

Библиотека хранит **любое верифицированное знание** — не только технические скиллы. Законы физики, паттерны поведения, логические правила, рабочие алгоритмы, проверенные промпты, бизнес-паттерны. Всё что доказало работоспособность или истинность через контур YMS-MMM.

**Метаданные = план в цифровом виде.** Локальная модель может развернуть набор метаданных-маячков в человекочитаемое описание через словарь библиотеки — для понимания как человеком, так и флагманом.

## 2. Трёхуровневая архитектура хранения

```
[ ХОЛОДНЫЕ РАСПРЕДЕЛЁННЫЕ ШАРДЫ (15 ГБ × N) ]
  Что: оригинальный UTF-8 / JSON / Код, сжатый zstd
  Формат: байт-в-байт, никакого квантования тела
  Приоритет: Hard-аксиома для флагмана, не обсуждается
  Зеркало: каждый шард дублируется на резервный хост
  Внутри файлов: вшитые гиперлинки [[EVO:символ]]
  на сопутствующие материалы — подтягиваются на лету
                    │
                    ▼  линк: символ → шард → ячейка
┌──────────────────────────────────────────────────────────┐
│  ЯЗЫК-БИБЛИОТЕКА (Когнитивная матрица / pgvector + RAM)   │
│  Строчные, надстрочные символы + цифровые значения        │
│  Фрактальное дерево: Наука → Раздел → Подраздел           │
│  Весь набор символов под план загружается сюда сразу      │
│  Последовательно вытягивается в оперативу по шагам плана  │
└──────────────────────────────────────────────────────────┘
                    │
                    ▼  горячий слой (TTL = время сессии)
[ REDIS — ГОРЯЧИЙ КЭШ (изолирован по юзеру) ]
  Что: часто применяемые символы текущей сессии
  Кто попадает: символы применяемые повторно в рамках плана
  TTL: время активности темы
  Схлопывание: послойное, с редактированием при улучшении
  Выгрузка: физическое удаление из RAM при закрытии сессии
```

## 3. Анатомия SCL-спрайта

Спрайт — высокоплотный символьный паспорт ветки знаний. Содержит четыре компонента:

```
[Уникальный ID символа] + [Векторный профиль интента] + [Линк → Шард → Ячейка] + [Рейтинг актуальности]
```

Спрайты живут в Redis (горячие, повторно используемые) и pgvector/RAM (весь набор под план). Локальная модель сканирует только этот слой — не читает архивы. Видит геометрию спрайтов и понимает какой резонирует с запросом.

## 4. Система нотации символа

```
[БАЗОВЫЙ_СИМВОЛ]^[надстрочный]_[подстрочный]_[номер]
```

| Уровень | Что | Пример |
|---------|-----|--------|
| Базовый символ | Греческая буква — макро-область из 32 | `τ` — технология |
| Надстрочный `^` | Специализация раздела | `^auto` — автоматизация |
| Подстрочный `_` | Среда / контекст / инструмент | `_zp` — ZennoPoster |
| Номер | Уникальный ID, автоинкремент | `_0047` |

Полный адрес: `τ^auto_zp_0047` — технология / автоматизация / ZennoPoster / знание №47

## 6. Лигатуры — ассимилированные знания

Лигатура — знание на стыке двух и более макро-областей. Создаётся при `confirmed_by >= 3`.

**Нотация:** `[Символ1 ⊕ Символ2]^[специализация]_[номер]`

```
[η ⊕ ε]^behav_buy_0003    — нейробиология + экономика / поведенческая экономика покупки
[τ ⊕ ε]^auto_crm_0011    — технология + экономика / автоматизация CRM
[ν ⊕ Φ]^align_verif_0001 — ИИ + философия / верификация истины в ИИ-системах
```

Лигатура не дублирует символы. Только уникальные точки пересечения при тройном подтверждении.

## 7. Двойная система рейтингов

### R_f — Частотный рейтинг
- Инкрементируется при каждом вызове: `R_f = R_f + 1`
- Инкрементируется при перезаписи ячейки: `R_f + 1`
- **Никогда не сбрасывается**
- Влияет на приоритет при поиске

### R_c — Контекстный рейтинг
- Вычисляется динамически в рамках текущей сессии
- **Объём не повышается** — индекс остаётся ультра-лёгким
- Не хранится как постоянная величина
- При закрытии сессии сбрасывается

### Ранжирование по слоям актуальности (1–5)

| Слой | Состояние | Хранение |
|------|-----------|----------|
| 1 | Активный шаг прямо сейчас | RAM — полная распаковка из zstd |
| 2 | Следующие шаги плана (загружены, ожидают) | pgvector/RAM — готовы к вытяжке |
| 3 | Часто повторяемые в сессии | Redis горячий кэш |
| 4 | Редко вызываемый | pgvector холодный индекс |
| 5 | Холодная лигатура (задача закрыта) | zstd на шарде, глубокий сон |

## 8. Алгоритм инициализации: Запрос Плана → Загрузка Набора → Последовательное Раскрытие

```
[ ПОЛЬЗОВАТЕЛЬ СТАВИТ ЗАДАЧУ ]
        │
        ▼
[ ЯДРО → ФЛАГМАНУ: «Какой у тебя пошаговый план решения?» ]
        │
        ▼
[ ФЛАГМАН ВЫДАЁТ ПОШАГОВЫЙ ПЛАН ]
  Шаг 1: ...  Шаг 2: ...  Шаг 3: ...
        │
        ▼
[ ЯДРО СКАНИРУЕТ МАЯЧКИ — ВЕСЬ НАБОР СИМВОЛОВ ПОД ПЛАН ]
  Все релевантные символы и лигатуры загружаются в pgvector/RAM
  сразу как единый набор под план.
  Часто применяемые в плане — дополнительно кэшируются в Redis.
        │
        ▼
[ ЯДРО → ФЛАГМАНУ: МЕТАДАННЫЕ КАК ПЛАН В ЦИФРОВОМ ВИДЕ ]
  Локальная модель разворачивает метаданные набора
  в человекочитаемое описание через словарь библиотеки:
  «Шаг 1: модуль авторизации ZennoPoster (HSR)»
  «Шаг 2: модуль парсинга аккаунтов»
  «Шаг 3: модуль крипто-шлюза через n8n»
  Флагман видит план, понимает что будет выдано, начинает работу.
        │
        ▼
[ ПОСЛЕДОВАТЕЛЬНОЕ РАСКРЫТИЕ — по мере выполнения шагов ]

  Флагман завершил Шаг 1 →
    ядро вытягивает символ Шага 1 из RAM в оперативу →
    декомпрессия zstd → детальная инструкция в контекст →
    флагман выполняет → локальная модель проверяет

  Флагман завершил Шаг 2 →
    ядро вытягивает символ Шага 2 → декомпрессия → выдача
    (если символ Шага 2 был в Redis — подача мгновенная)

  ...и так далее по последовательности из метаданных набора.

  Смещение фокуса: общий план → каждый шаг детально,
  только когда до него доходит очередь.
```

**Ключевое:** все символы под план загружаются в RAM/pgvector **сразу** при инициализации — но детально раскрывается каждый только после того как предыдущий шаг выполнен. Последовательность определяется метаданными набора.

## 9. Три сценария при поиске

### Сценарий А — Комплексная лигатура есть

Найдена лигатура покрывающая весь план целиком.

1. Ядро выдаёт флагману лигатуру как основу
2. Фрактально находит микро-символы из которых она состоит
3. Загружает весь набор в RAM/pgvector
4. Раскрывает каждый символ по очереди по мере выполнения шагов
5. Флагман получает полный укомплектованный картридж

### Сценарий Б — Комплексной лигатуры нет, но есть отдельные символы

Модульная сборка новой лигатуры на месте:

```
[ ЯДРО НАХОДИТ ОТДЕЛЬНЫЕ СИМВОЛЫ ПОД КАЖДЫЙ ШАГ ПЛАНА ]
        │
        ▼
[ ЯДРО СОБИРАЕТ НОВУЮ ЛИГАТУРУ ИЗ ЭТИХ СИМВОЛОВ ]
  Создаётся прямо сейчас под конкретный план
  В метаданных: сигнатуры разделов + описание чем отличается
  от соседних лигатур по векторизации
        │
        ▼
[ НОВАЯ ЛИГАТУРА ЗАПИСЫВАЕТСЯ В ЯЗЫК-БИБЛИОТЕКУ ]
  Вектор в pgvector. zstd тела — асинхронно на шард.
        │
        ▼
[ ЛИГАТУРА РАСКЛАДЫВАЕТСЯ НА СОСТАВНЫЕ СИМВОЛЫ ]
  Ядро удерживает лигатуру в памяти сессии
  и последовательно раскрывает её символы по шагам плана
        │
        ▼
[ ВЫПОЛНЕНИЕ ПО ОЧЕРЕДИ С ПРОВЕРКОЙ ЭФФЕКТИВНОСТИ ]
  После каждого шага — локальная модель проверяет результат.
  Если найдено более эффективное решение на каком-то шаге →
  символ этого шага обновляется (Тип А/Б),
  лигатура перезаписывается с учётом улучшения.
```

**Директива флагману при частичном совпадении:**

```
[EVO DIRECTIVE — PARTIAL MATCH]

Стопроцентного решения под твои вводные нет.
Я успешно решал смежную задачу вот с таким набором: [Инструкция_X].
В твоих вводных заданы новые переменные.

Твоя задача:
1. Изучи выданный базис
2. Найди или адаптируй решение под свои вводные
3. Проведи тесты внутри песочницы
4. Выдай работающее решение
5. Покажи мне результат для проверки

После проверки я сам сформулирую логику, создам символ
и запишу знание в базу.
```

### Сценарий В — Знаний нет совсем

Сначала — поиск по имеющимся символам на предмет смежного.
Если смежного недостаточно — поиск во внешних источниках.

**Приоритет внешних источников:**
1. GitHub (открытые репозитории, высокий рейтинг, актуальные коммиты)
2. Официальная документация инструментов (npm, PyPI, docs)
3. Маркетплейсы скиллов (n8n Templates, ZennoPoster форум)
4. Открытые источники с подтверждённым рейтингом сообщества

**Директива флагману при отсутствии знания:**

```
[EVO DIRECTIVE — KNOWLEDGE GAP]

В текущей базе знаний нет решения под эту задачу.

Твоя задача:
1. Найди точное решение в внешних источниках:
   - GitHub (высокорейтинговые репозитории)
   - Официальная документация
   - Проверенные маркетплейсы скиллов
2. Соотнеси найденное с имеющимися символами EVO
3. Выполни задачу пользователя
4. Проведи тесты — докажи работоспособность
5. Выдай верифицированный результат

Я заберу результат, проверю и запишу в базу навсегда.
```

## 11. Тип А — Универсализация (перезапись)

Флагман улучшил решение (быстрее, актуальнее, безопаснее). Работает в тех же обстоятельствах — только лучше.

**Действия:**
1. Тело zstd-ячейки на шарде физически заменяется новым
2. `R_f += 1`
3. Переходный маяк: `было [старый метод] → стало [новый метод]`
4. Дублей не создавать

**Хук-допрос после записи:**
> **«...или есть что-то ещё новее?»**

Флагман проверяет по актуальным параметрическим данным. Если вышло обновление — цикл повторяется. Ядро удерживается на пике технологий времени автоматически.

---

## 12. Тип Б — Дифференциация (новый символ)

Кардинально иные инструменты или обстоятельства. Нельзя слить в одну ячейку.

**Действия:**
1. Новый символ рядом с тематическими инструментами
2. `evolved_from` = ID родительского
3. Метаданные: чем отличается от соседних по векторизации
4. В теле — гиперлинк на родительский: `[[EVO:родительский_id]]`

## 13. Алгоритм схлопывания сессии (с редактированием)

Схлопывание — не просто удаление. Если в ходе сессии найдено более эффективное решение — символы и лигатуры **редактируются перед выгрузкой**.

```
ТРИГГЕР: тема исчерпана / сессия закрывается
        │
        ▼
АСИНХРОННАЯ ЗАПИСЬ (новое знание если есть)
  zstd + шард + pgvector — в фоне, пользователь не ждёт
        │
        ▼
РЕДАКТИРОВАНИЕ ПЕРЕД СХЛОПЫВАНИЕМ
  Для каждого активного символа/лигатуры:
  Если в ходе сессии найдено более эффективное решение →
    обновить тело ячейки (Тип А или Тип Б)
    зафиксировать переходный маяк
    R_f += 1
        │
        ▼
СХЛОПЫВАНИЕ В ОБРАТНОМ ПОРЯДКЕ АКТИВАЦИИ

  Слой 1 (последние активные шаги) →
    редакция если нужна → физическое удаление из RAM

  Слой 2 (символы шагов плана, ожидавшие очереди) →
    редакция если нужна → физическое удаление из pgvector/RAM

  Слой 3 (Redis горячий кэш — повторно используемые) →
    редакция если нужна → физическое удаление из Redis

  Данные остаются только в холодном слое (шарды)
        │
        ▼
ИЗОЛЯЦИЯ ЗАВЕРШЕНА
  Данные этого юзера физически отсутствуют в оперативной памяти
  Данные других юзеров не затронуты
```

## 14. Полный алгоритм формирования нового символа (7 шагов)

### Шаг 1 — Макро-корень
К какой из 32 областей знание относится первично?
- Одна → символ
- Несколько + `confirmed_by >= 3` → лигатура (Шаг 1б)

**Шаг 1б:** Определить корни. Проверить что комбинации нет. Если нет — создать.

### Шаг 2 — Надстрочный (специализация)
Если нет — создать новый, зафиксировать.

## 16. Правила (нарушение = Критическая Ошибка YMS-MMM)

1. **32 базовых символа неизменяемы.** Новые корни не создаются.
2. **R_f только инкрементируется.** Никогда не сбрасывается. Перезапись = `R_f + 1`.
3. **R_c не раздувается.** Контекстный вес статичен.
4. **Тело хранится в оригинале.** Только zstd, байт-в-байт.
5. **Хронология неприкосновенна.** `evolved_from`, `evolution_note`, `last_updated` не удаляются.
6. **Дублей не создавать.** Проверка similarity обязательна до записи.
7. **Лигатура только при `confirmed_by >= 3`.**
8. **Гиперлинки вшиваются в тело.** Параллельная подгрузка при декомпрессии.
9. **Хук-допрос обязателен** после каждой успешной записи.
10. **Набор загружается сразу, раскрывается по очереди.** Все символы под план — в RAM/pgvector сразу. Детальная распаковка каждого — только когда предыдущий шаг выполнен. Последовательность из `sequence_position` в метаданных.
11. **Метаданные = план в цифровом виде.** Локальная модель разворачивает маячки в человекочитаемое описание через словарь библиотеки.
12. **Redis для горячих повторяемых.** Символы применяемые повторно в рамках одной сессии кэшируются в Redis. Весь набор под план — в pgvector/RAM.
13. **Схлопывание с редактированием.** Перед выгрузкой из памяти — обновить символы если найдено более эффективное решение. Затем физически удалить послойно в обратном порядке активации.
14. **Изоляция сессий.** Данные разных пользователей не пересекаются в оперативной памяти.
15. **Флагман не знает внутренней кухни.** Шарды, zstd, структура символов — под капотом.
16. **При отсутствии знания — поиск внешний.** Приоритет: GitHub высокорейтинговые репо → официальная документация → маркетплейсы скиллов. Найденное верифицируется через YMS-MMM перед записью.
17. **Лигатура собранная на месте — записывается в библиотеку.** Затем раскладывается на составные символы и выполняется по шагам. После выполнения — проверяется на эффективность и при необходимости перезаписывается.
