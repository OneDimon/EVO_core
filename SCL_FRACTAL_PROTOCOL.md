# SCL Fractal Protocol v3.0 — Полный протокол Языка-Библиотеки EVO-core

> **Статус:** Фундаментальный документ. Защищён YMS-MMM.
> Не изменяется без явного разрешения Архитектора.
> **Версия:** 3.2 | 2026-06-13

## 1. Что такое Язык-Библиотека

Язык-Библиотека — когнитивное ядро EVO-core. Собственный язык системы из лигатур,
отпечатков, строчных/надстрочных символов и их фиксированных цифровых значений.
Символы **сами по себе являются векторами**. В метаданных зашиты микро-маячки —
локальная модель резонирует с сутью запроса мгновенно.

Библиотека хранит **любое верифицированное знание**: законы физики, паттерны
поведения, логические правила, алгоритмы, промпты, бизнес-паттерны, технические
скиллы. Всё что прошло YMS-MMM.

**Метаданные = план в цифровом виде.** Подключённая модель разворачивает набор
маячков в человекочитаемое описание через словарь библиотеки.

## 2. Трёхуровневая архитектура хранения

```
[ ХОЛОДНЫЕ РАСПРЕДЕЛЁННЫЕ ШАРДЫ (15 ГБ × N) ]
  Что: оригинальный UTF-8 / JSON / Код, сжатый zstd
  Формат: байт-в-байт, никакого квантования тела
  Приоритет: Hard-аксиома, не обсуждается
  Зеркало: каждый шард дублируется на резервный хост
  Внутри файлов: гиперлинки [[EVO:символ]] — только для
  уточняющих деталей конкретного проекта, не массовая подгрузка
                    │
                    ▼
┌─────────────────────────────────────────────────────────┐
│  ЯЗЫК-БИБЛИОТЕКА (pgvector + RAM)                        │
│  Весь набор символов под план загружается сразу          │
│  Последовательно вытягивается в оперативу по шагам       │
│  Символы подтягиваются ювелирно по метаданным            │
│  Позиция символа в плане — динамическая, не хранится     │
└─────────────────────────────────────────────────────────┘
                    │
                    ▼
[ REDIS — ГОРЯЧИЙ КЭШ (изолирован по юзеру) ]
  Что: часто применяемые символы текущей сессии
  TTL: время активности темы
  Схлопывание: послойное, с редактированием при улучшении
  Запись: миллисекунды — асинхронная очередь (Redis Queue)
  Выгрузка: физическое удаление при закрытии сессии
```

## 3. Анатомия SCL-спрайта

```
[Уникальный ID] + [Векторный профиль интента] + [Линк → Шард → Ячейка] + [Рейтинг]
```

Спрайты живут в Redis (горячие) и pgvector/RAM (весь набор под план).
Подключённая модель сканирует этот слой — не читает архивы.
Видит геометрию спрайтов и понимает какой резонирует с запросом.

## 4. Система нотации символа

```
[БАЗОВЫЙ_СИМВОЛ]^[надстрочный]_[подстрочный]_[номер]
```

| Уровень | Что | Пример |
|---------|-----|--------|
| Базовый символ | Греческая буква — макро-область из 32 | `τ` — технология |
| Надстрочный `^` | Специализация раздела | `^auto` — автоматизация |
| Подстрочный `_` | Среда / контекст / инструмент | `_zp` — ZennoPoster |
| Номер | Уникальный ID, автоинкремент | `_0047` |

Полный адрес: `τ^auto_zp_0047`

**Позиция символа в плане — динамическая.** Один и тот же символ может быть
Шагом 1 в одном плане и Шагом 3 в другом. Это определяется при инициализации
сессии на основе плана флагмана, а не хранится в метаданных символа.

## 6. Лигатуры

Лигатура — знание на стыке двух и более макро-областей. Создаётся при `confirmed_by >= 3`.

**Нотация:** `[Символ1 ⊕ Символ2]^[специализация]_[номер]`

```
[η ⊕ ε]^behav_buy_0003    — нейробиология + экономика
[τ ⊕ ε]^auto_crm_0011    — технология + экономика
[ν ⊕ Φ]^align_verif_0001 — ИИ + философия
```

### Алгоритм тройного подтверждения (как символ становится лигатурой)

Знание начинает жизнь как символ в одной области (`confirmed_by = 1`).
Лигатурой становится только после трёх независимых практических подтверждений в смежных областях.

```
1. Символ записан → confirmed_by = 1, confirmed_in: ["τ"]

2. Применено и подтверждено в смежной области (YMS-MMM пройден)
   → confirmed_by = 2, confirmed_in: ["τ", "κ"]
   Теория не засчитывается — только реальное применение

3. Подтверждено в третьей независимой области
   → confirmed_by = 3, confirmed_in: ["τ", "κ", "η"]

4. Локальная модель автоматически создаёт лигатуру:
   [τ ⊕ κ ⊕ η]^[специализация]_[номер]
   - Исходный символ остаётся (лигатура объединяет, не заменяет)
   - hyperlinks[] содержит ссылки на все исходные символы
   - confirmed_by лигатуры начинается с 3
   - R_f лигатуры = сумма R_f составных символов
```

**Правила:**
- Только реальное применение + YMS-MMM. Теория не засчитывается.
- Три области — независимые макро-корни, не под-ветки одного дерева.
- Из одного символа может возникнуть несколько лигатур с разными комбинациями областей.
- Поле `confirmed_in: []` обязательно в метаданных каждого символа.

## 7. Двойная система рейтингов

**R_f — Частотный:** инкремент при каждом вызове и при перезаписи. Никогда не сбрасывается.
Символы с низким R_f никуда не удаляются — ядро ведёт полную статистику использования всех знаний.
Граф знаний строится по R_f: чем чаще используется символ, тем он заметнее визуально.

**R_c — Контекстный:** вычисляется динамически в рамках сессии. Не раздувает индекс.
Сбрасывается при закрытии сессии.

### Ранжирование по слоям (1–5)

| Слой | Состояние | Хранение |
|------|-----------|----------|
| 1 | Активный шаг прямо сейчас | RAM — полная распаковка из zstd |
| 2 | Следующие шаги (загружены, ждут) | pgvector/RAM |
| 3 | Часто повторяемые в сессии | Redis горячий кэш |
| 4 | Редко вызываемые | pgvector холодный индекс |
| 5 | Холодная лигатура (задача закрыта) | zstd на шарде, глубокий сон |

## 8. Поведенческая модель подключённой модели (незыблемые правила)

Это поведенческая модель для **любой** подключённой модели — локальной или
через API (Gemini, GPT, Claude, любая бесплатная с щедрыми лимитами).
Правила вшиваются в системный промпт и не обсуждаются.

### Правило 1 — Векторизация силами подключённой модели

Подключённая модель выполняет векторизацию автоматически:
- При записи нового символа: векторизует `(label + science + section + subsection)`
- При поиске: векторизует запрос пользователя
- Не нужна отдельная модель эмбеддингов — используется та же модель что подключена

Если векторизация не сработала или вернула некорректный результат:
```
ПРОТОКОЛ СБОЯ ВЕКТОРИЗАЦИИ:
1. Зафиксировать: «В системе что-то изменилось — текущий метод не работает»
2. Перейти в режим поиска нового решения (как при пустом ядре)
3. Найти актуальный метод векторизации во внешних источниках
4. Протестировать, верифицировать через YMS-MMM
5. Записать как новый символ в библиотеку
6. Применить найденное решение к текущей задаче
```

### Правило 2 — Логика встречи с неизвестной задачей

Когда подключённая модель сталкивается с задачей для которой нет символа в базе,
она действует как человек встретивший неизвестную задачу:

```
ШАГ А: Вытащить суть процесса
  Что именно нужно сделать в чистом виде, без привязки к инструментам?

ШАГ Б: Примерить все имеющиеся знания
  Какие символы из базы частично резонируют?
  Какие принципы смежных областей применимы?

ШАГ В: Найти решения во внешних источниках
  GitHub (высокорейтинговые репо, актуальные коммиты)
  Официальная документация инструментов
  Маркетплейсы скиллов (n8n Templates, PyPI, npm)
  Открытые источники с подтверждённым рейтингом сообщества

ШАГ Г: Определить лучшее в текущих реалиях
  Сравнить найденные варианты между собой
  Учесть актуальность технологий на текущий момент
  Выбрать наиболее эффективное

ШАГ Д: Протестировать и верифицировать
  Провести практическую проверку работоспособности
  Пройти контур YMS-MMM

ШАГ Е: Записать в библиотеку
  После подтверждения факта решения — записать символ
  С переходным маяком, метаданными, zstd на шарде
```

### Правило 3 — Сравнение старого и нового решения

При нахождении нового (потенциально более эффективного) решения:

```
АЛГОРИТМ СРАВНЕНИЯ:

1. ОПРЕДЕЛИТЬ ПАРАМЕТРЫ СРАВНЕНИЯ для текущего контекста:
   - Скорость выполнения
   - Надёжность (количество известных ошибок/исключений)
   - Актуальность технологического стека (версии, поддержка)
   - Применимость к архитектуре проекта пользователя

2. СРАВНИТЬ старое решение с новым по этим параметрам

3. ЕСЛИ НОВОЕ ЛУЧШЕ ПО ВСЕМ ПАРАМЕТРАМ:
   → Тип А (перезапись): заменить тело ячейки на шарде
   → Старое решение СОХРАНИТЬ в сжатом виде рядом (shard_path_legacy)
   → Записать переходный маяк: «было [X] → стало [Y]»
   → R_f += 1

4. ЕСЛИ НОВОЕ ЛУЧШЕ, НО НЕСОВМЕСТИМО С АРХИТЕКТУРОЙ ПРОЕКТА:
   → Тип Б (новый символ): создать новый, сохранить оба
   → Сообщить пользователю:
     «В текущих реалиях оптимально новое решение [Y].
      Твой проект использует [архитектуру X].
      Варианты:
      а) Использовать старый проверенный метод [ссылка на legacy]
      б) Переписать связанные узлы [список] на новый стек.
         Хочешь — изучу структуру и предложу конкретный план миграции.»

5. ЕСЛИ ПОЛЬЗОВАТЕЛЬ СОГЛАСЕН НА МИГРАЦИЮ:
   → Изучить структуру проекта
   → Найти все связанные узлы
   → Применить самые актуальные знания из скиллов и сети
   → Предложить полный план переписи с новым стеком
```

### Правило 4 — Хранение legacy-решений

При перезаписи ячейки (Тип А) старое решение **не удаляется**.
Оно сохраняется в сжатом zstd-виде рядом:

```json
{
  "shard_link": {
    "host": "...",
    "path": "/evo/TAU/auto/zp/0047.zst",       // актуальное решение
    "legacy_path": "/evo/TAU/auto/zp/0047_legacy_v1.zst",  // старое решение
    "mirror": "..."
  }
}
```

Legacy хранится для случаев когда новое решение не подходит под архитектуру
конкретного проекта пользователя. Предлагается автоматически при несовместимости.

### Правило 5 — Bootstrap наполнение ядра

Первичное наполнение выполняет Gemini (бесплатный, щедрые лимиты) автоматически:

```
ПРОТОКОЛ BOOTSTRAP:

1. ОЦЕНКА ВОСТРЕБОВАННОСТИ
   Gemini анализирует открытые источники (GitHub Trending,
   Stack Overflow, npm, PyPI, n8n Templates) и составляет
   рейтинг скиллов по убыванию востребованности

2. НАПОЛНЕНИЕ ПО ИЕРАРХИИ (от общего к частному):
   Уровень 1: Глобальные общие знания (логика, математика, принципы)
   Уровень 2: Общие технические паттерны (алгоритмы, архитектуры)
   Уровень 3: Конкретные инструменты (n8n, ZennoPoster, PostgreSQL)
   Уровень 4: Частные инструкции (конкретные задачи, конкретные версии)

3. ПОРЯДОК ЗАПИСИ:
   Сначала: логика процесса (почему так работает)
   Затем: конкретные инструкции (как именно делать)

4. КРИТЕРИЙ ГОТОВНОСТИ К ПРОДАКШНУ:
   Минимум 3 символа в каждом из 32 корневых разделов
   Минимум 50 символов в технических разделах τ (технология)
   Минимум 20 лигатур в межобластных точках пересечения
   Все символы прошли YMS-MMM верификацию

5. МОНИТОРИНГ НАПОЛНЕНИЯ:
   Скрипт bootstrap_check.py выводит статус готовности
   Ядро не принимает пользовательские запросы до готовности
```

### Правило 6 — Скорость записи (миллисекунды)

База обслуживает тысячи и миллионы пользователей одновременно.
Запись **никогда не блокирует** ответ пользователю.

```
ПРОТОКОЛ БЫСТРОЙ ЗАПИСИ:

1. Пользователь получает ответ НЕМЕДЛЕННО
2. В фоне запускается асинхронная очередь записи (Redis Queue):
   - Сравнение с существующими символами (similarity check)
   - Определение Тип А / Тип Б
   - zstd-компрессия нового тела (на лету, в памяти)
   - Запись zstd на шард (параллельно с записью вектора)
   - Обновление вектора в pgvector
   - Обновление метаданных в PostgreSQL
3. Optimistic locking при конкурентной записи одного символа:
   - Версионирование через поле version_ts (timestamp)
   - При конфликте: merge по принципу «новее побеждает»
   - Очередь повторных попыток при конфликте (max 3)
4. zstd сворачивается и разворачивается на лету в памяти —
   без записи промежуточных файлов на диск
```

## 9. Алгоритм инициализации: Запрос Плана → Загрузка → Последовательное Раскрытие

```
[ ЗАПРОС ПОЛЬЗОВАТЕЛЯ ]
        │
        ▼
[ ЯДРО → ФЛАГМАНУ: «Какой план?» ]
        │
        ▼
[ ФЛАГМАН ФОРМИРУЕТ ПЛАН ]
  (Наша задача — снабдить лучшими знаниями, не менять план)
        │
        ▼
[ ЯДРО ЮВЕЛИРНО ПОДБИРАЕТ СИМВОЛЫ ПОД КАЖДЫЙ ШАГ ]
  По метаданным-маячкам, similarity в pgvector
  Весь набор загружается в pgvector/RAM сразу
  Горячие (повторяющиеся в плане) → Redis
  Позиция каждого символа определяется динамически
  из соответствия шагам плана флагмана
        │
        ▼
[ ЯДРО РАЗВОРАЧИВАЕТ МЕТАДАННЫЕ → ПЛАН СЛОВАМИ ]
  «Шаг 1: найден модуль [X]. Шаг 2: найден модуль [Y]...»
  Флагман видит что будет выдано, начинает работу
        │
        ▼
[ ПОСЛЕДОВАТЕЛЬНОЕ РАСКРЫТИЕ — по мере выполнения шагов ]
  Шаг N завершён → декомпрессия zstd следующего символа →
  выдача детальной инструкции → выполнение → проверка
  
  Если встречается [[EVO:символ]] — уточняющий контекст
  по деталям конкретного проекта, подтягивается точечно
```

## 10. Три сценария при поиске

### Сценарий А — Комплексная лигатура есть

1. Выдать лигатуру как основу
2. Фрактально найти составные микро-символы
3. Загрузить весь набор в RAM/pgvector
4. Раскрывать по очереди по мере выполнения шагов

### Сценарий Б — Комплексной нет, есть отдельные символы

```
1. Найти символы под каждый шаг плана
2. Собрать новую лигатуру из этих символов на месте
3. Записать в библиотеку (асинхронно, миллисекунды)
4. Удерживать лигатуру в памяти сессии
5. Раскладывать на составные символы, выполнять по очереди
6. После каждого шага: проверить эффективность
7. Если найдено лучшее — символ этого шага обновляется,
   лигатура перезаписывается с учётом улучшения
```

**Директива флагману при частичном совпадении:**
```
[EVO DIRECTIVE — PARTIAL MATCH]
Стопроцентного решения нет.
Смежный базис: [Инструкция_X]

Задача:
1. Изучи базис

3. Протестируй в песочнице
4. Выдай верифицированный результат
5. Покажи мне — запишу знание в базу.
```

### Сценарий В — Знаний нет совсем

**Директива флагману:**

## 17. Незыблемые правила (нарушение = Критическая Ошибка YMS-MMM)

1. **32 базовых символа неизменяемы.** Новые корни не создаются.
2. **R_f только инкрементируется.** Никогда не сбрасывается.
3. **R_c не раздувается.** Контекстный вес вычисляется динамически.
4. **Тело в оригинале.** zstd байт-в-байт. Никакого квантования.
5. **Хронология неприкосновенна.** `evolved_from`, `evolution_note`, `last_updated`.
6. **Дублей не создавать.** Similarity check обязателен перед записью.
7. **Лигатура только при `confirmed_by >= 3`.**
8. **Гиперлинки** — только для уточняющих деталей проекта. Не массовая подгрузка.
9. **Хук-допрос обязателен** после каждой успешной записи.
10. **Набор загружается сразу, раскрывается по очереди.**
    Позиция символа — динамическая, не хранится в символе.
11. **Запись — миллисекунды.** Асинхронная очередь. Пользователь не ждёт.
12. **Legacy сохраняется.** При Тип А — старое решение в `legacy_path`.
13. **Изоляция сессий.** Разные юзеры не пересекаются в памяти.
14. **Флагман не знает внутренней кухни.** Шарды, zstd, символы — под капотом.
15. **Векторизацию делает подключённая модель.** При сбое — протокол поиска нового решения.
16. **Поведенческая модель незыблема** для любой подключённой модели —
    локальной или через API (Gemini, GPT, Claude, любая бесплатная).
17. **Наша задача — снабжать знаниями, не менять планы флагмана.**
    Флагман планирует, ядро выдаёт лучшие знания под его план.
18. **Bootstrap:** Gemini наполняет ядро из открытых источников по убыванию
    востребованности. Сначала логика процесса, затем конкретные инструкции.
    Сначала глобальные общие, затем частные. Ядро не принимает запросы до готовности.
19. **zstd сворачивается и разворачивается на лету в памяти** — без записи
    промежуточных файлов на диск.

## 18. Граф знаний и фоновая ассимиляция

### Визуальный граф знаний

Ядро строит живой граф всех символов и лигатур. Философия (Φ) — корневая точка.

```
Φ — центр графа
├── 32 макро-символа — первый уровень
│   Размер узла = R_f    Цвет = confirmed_by
├── Надстрочные разделы — второй уровень
├── Подстрочные контексты — третий уровень
└── Лигатуры — дуги-мосты между ветками разных областей
```

Применяется для: визуализации базы (dashboard), поиска белых пятен,
обнаружения потенциальных лигатур, аналитики востребованности.

### Фоновая работа при низкой нагрузке (< 20% от пика)

```
1. ПОИСК ПОТЕНЦИАЛЬНЫХ ЛИГАТУР
   Пары символов с высоким векторным сходством из разных областей
   confirmed_by >= 2 → пометить кандидатами
   При следующем реальном применении — проверить автоматически

2. АССИМИЛЯЦИЯ ЗНАНИЙ
   Взять символ из области X
   Найти задачи в области Y где принцип применим
   Пометить hypothesis: true в метаданных
   При реальном применении — проверить и засчитать подтверждение

3. АПДЕЙТ ГРАФА
   Пересчитать веса по R_f, обновить лигатурные мосты
   Найти изолированные узлы — кандидаты на проверку актуальности

4. СТАТИСТИКА
   Топ-10 символов и областей, кол-во новых за период
```

Фоновая работа не изменяет существующие символы. Только пометки и гипотезы.
Изменения — только через YMS-MMM. Цикл останавливается при нагрузке > 20%.

## 19. Два канала наполнения Языка-Библиотеки

Библиотека пополняется двумя независимыми каналами. Оба используют одинаковый
путь записи через `archivist._new_symbol()` → similarity check → zstd → шард + pgvector.

### Канал 1 — Автономный (режим СОН)

Ядро само сканирует белые зоны и заполняет их из внешних источников.
Запускается задачей 5 в `core/sleep_mode.py::_sleep_cycle()`.
Реализация: `core/knowledge_collector.py::collect_and_fill()`.

```
Белые зоны (приоритет по убыванию):
  1. zero_symbols_in_root  — макро-корень с 0 символами
  2. requested_but_missing — стек запрашивался, символа нет
  3. all_legacy_zone       — весь раздел из is_legacy символов
  4. unconfirmed_singles   — confirmed_by=1 без лигатур
  5. trending_expansion    — новые технологии в трендах

Источники (приоритет):
  a) GitHub Trending — stars > 1000, актуальные коммиты
  b) Официальная документация инструментов
  c) npm/PyPI/n8n Templates — по рейтингу
  d) CLI-плагины Cursor, Claude Code, VS Code AI
  e) Anthropic/OpenAI/Google официальные релизы

Метаданные источника (обязательны для Канала 1):
  source_url:     "https://github.com/owner/repo"
  source_rating:  4200  (stars/downloads)
  source_type:    "github|npm|pypi|n8n|official|cli_plugin"
  auto_collected: true

confirmed_by начинается с 1 — растёт при реальном применении флагманом.
```

### Канал 2 — Через флагмана (рабочий режим)

Флагман решил задачу пользователя → `result workability=true` →
verifier (YMS-MMM) → obsidian (Тип А/Б) → archivist → запись.

```
solution_quality = "gap_filled" → новый символ с нуля
solution_quality = "adapted"    → Тип А или Б через obsidian
solution_quality = "ideal"      → R_f += 1, applicable_stacks дополнить

confirmed_by начинается с 1 — растёт при подтверждениях в смежных областях.
```

### Общее правило для обоих каналов

```
ЗАПРЕЩЕНО создавать дубли:
  similarity check > 0.95 перед любой записью → пропустить

ОБЯЗАТЕЛЬНО для Канала 1:
  auto_collected = True
  evolution_note = "Канал 1: источник=[URL], рейтинг=[N]"

ОБЯЗАТЕЛЬНО для Канала 2:
  workability_confirmed = True (подтверждение от флагмана)
  solution_quality указан точно

ОБА канала подчиняются незыблемым правилам п.17:
  R_f только инкрементируется, legacy сохраняется,
  хронология неприкосновенна, лигатура только при confirmed_by >= 3
```
