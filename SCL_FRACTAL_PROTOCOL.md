# SCL Fractal Protocol — Инструкция формирования Языка-Библиотеки

> **Статус:** Фундаментальный документ. Не изменяется без явного разрешения Архитектора.  
> **Назначение:** Определяет как система (автоматически или вручную) формирует символы, лигатуры, фракталы и записывает любое верифицированное знание в Язык-Библиотеку EVO-core.

---

## Принцип первый: Что записывается в библиотеку

Библиотека хранит **любое верифицированное знание** — не только технические скиллы. Верифицированное означает: прошло через контур YMS-MMM, доказало работоспособность или истинность.

Примеры того что записывается:
- Рабочий алгоритм авторизации в ZennoPoster
- Закон сохранения импульса (физика, подтверждён в 3+ областях)
- Паттерн поведения покупателя при дефиците (психология + экономика)
- Оптимальная структура n8n воркфлоу для крипто-шлюза
- Логическое правило дедукции из формальной логики
- Проверенный промпт для верификации кода

Граница одна: **знание должно быть верифицировано**. Гипотезы и черновики хранятся в сессионном кэше, в библиотеку не идут.

## Принцип второй: Структура символа как адресной единицы

Каждая единица в библиотеке (символ или лигатура) — это адрес в пространстве знаний. Адрес состоит из четырёх уровней нотации:

```
[БАЗОВЫЙ СИМВОЛ][подстрочный_контекст]^[надстрочная_специализация]_[порядковый_номер]
```

### Базовый символ
Греческая буква — макро-область знаний. Неизменяемый корень. Их 32.

### Надстрочный индекс (^)
Раздел внутри макро-области. Конкретизирует специализацию.
Пример: τ^web — технология веб-разработки. τ^auto — технология автоматизации.

### Подстрочный индекс (_)
Контекст применения или среда. Уточняет условия.
Пример: τ^auto_zp — автоматизация в ZennoPoster. τ^auto_n8n — автоматизация в n8n.

### Порядковый номер
Уникальный ID конкретного знания внутри подраздела. Инкрементируется автоматически.
Пример: τ^auto_zp_0047 — сорок седьмое верифицированное знание по автоматизации в ZennoPoster.

### Степень (рейтинг)
Отражает количество подтверждений и вызовов. Только инкрементируется, никогда не сбрасывается.
Обозначается в метаданных как `rating_frequency` и `confirmed_by` (сколько смежных областей подтвердили).

### Полный пример адреса
```
τ^auto_zp_0047  — технология / автоматизация / ZennoPoster / скилл №47
β^neuro_cogn_0012 — биология / нейробиология / когнитивистика / знание №12
Φ^logic_ded_0003 — философия / логика / дедукция / правило №3
```

## Принцип четвёртый: Лигатуры — ассимилированные знания

**Лигатура** — это символ на стыке двух или более макро-областей. Создаётся когда знание подтверждено в 3 и более смежных науках одновременно и не принадлежит только одной из них.

### Нотация лигатуры
```
[Символ1 ⊕ Символ2]^[специализация]_[номер]
```

Пример:
```
[η ⊕ ε]^behav_buy_0003 — нейробиология + экономика / поведенческая экономика покупки / знание №3
[τ ⊕ ε]^auto_crm_0011 — технология + экономика / автоматизация CRM / знание №11
[ν ⊕ Φ]^align_verif_0001 — ИИ + философия / верификация истины в ИИ-системах / знание №1
```

### Условия создания лигатуры
1. Знание проверено в 3+ смежных областях (confirmed_by >= 3)
2. Нельзя однозначно отнести только к одному базовому символу
3. Ранее такой лигатуры в библиотеке не существовало

Лигатура не дублирует символы — она создаётся только для уникальных точек пересечения.

## Принцип пятый: Алгоритм фрактального формирования нового знания

Это инструкция которой следует система (и человек) при появлении нового верифицированного знания. Применяется автоматически после успешного прохождения YMS-MMM.

### Шаг 1 — Определить макро-корень

Задать вопрос: **к какой из 32 областей относится это знание прежде всего?**

Если однозначно к одной — это символ. Выбрать базовый символ из таблицы.
Если к двум и более, и подтверждено в 3+ областях — это кандидат на лигатуру. Перейти к Шагу 1б.

**Шаг 1б (для лигатуры):** Определить два-три корневых символа. Проверить что такой комбинации ещё нет в библиотеке. Если нет — создать новую лигатуру. Если есть — добавить знание внутрь существующей.

### Шаг 2 — Определить надстрочный индекс (специализация)

Спуститься на один уровень вниз: **какой раздел внутри этой области?**

Примеры надстрочных для τ (технология):
- `^web` — веб-разработка
- `^auto` — автоматизация
- `^db` — базы данных
- `^infra` — инфраструктура
- `^sec` — безопасность систем
- `^api` — интеграции и API

Если нужного надстрочного не существует — создать новый. Зафиксировать в метаданных символа.

### Шаг 3 — Определить подстрочный индекс (контекст)

Спуститься ещё на уровень: **в какой конкретной среде, инструменте или условии?**

Примеры подстрочных для τ^auto (автоматизация):
- `_zp` — ZennoPoster
- `_n8n` — n8n
- `_py` — Python скрипты
- `_rs` — Rust
- `_gs` — Google Sheets

Если нужного подстрочного не существует — создать новый.

### Шаг 4 — Присвоить порядковый номер

Найти в библиотеке максимальный номер для данного подраздела (`τ^auto_zp_XXXX`) и взять следующий: `max + 1`.

Если подраздел новый — начать с `0001`.

### Шаг 5 — Заполнить метаданные

```json
{
  "id": "τ^auto_zp_0047",
  "label": "ZennoPoster — авторизация через куки без пароля (HSR)",
  "science": "Технология",
  "section": "Автоматизация",
  "subsection": "ZennoPoster",
  "rating_frequency": 1,
  "confirmed_by": 1,
  "evolved_from": "τ^auto_zp_0031",
  "evolution_note": "Переход с Genshin Impact → Honkai Star Rail; крипто-шлюз вместо карт",
  "last_updated": "2026-06-03T00:00:00Z",
  "hyperlinks": ["τ^auto_n8n_0033", "ε^pay_crypto_0021"],
  "shard_link": {
    "host": "shard-mirror-04.free-cloud.io",
    "path": "/evo/TAU/auto/zp/0047.zst",
    "mirror": "shard-mirror-07.free-cloud.io"
  }
}
```

### Шаг 6 — Упаковать тело знания

Взять точный верифицированный текст (код, инструкцию, формулировку закона, паттерн). Сжать `zstd`. Загрузить на шард. Записать путь в `shard_link`.

Внутри текста вшить гиперлинки на связанные знания в формате:
```
[[EVO:τ^auto_n8n_0033]]
[[EVO:ε^pay_crypto_0021:step_3]]
```

### Шаг 7 — Проверка на дублирование

Перед записью: найти в pgvector ближайшие символы по cosine similarity.

- Similarity > 0.95 с существующим: **не создавать новый**, обновить существующий (Тип А — перезапись, рейтинг +1).
- Similarity 0.75–0.95: **создать новый**, в метаданных указать `evolved_from` на ближайший.
- Similarity < 0.75: **новый независимый символ**.

## Принцип шестой: Алгоритм автоматического формирования (для системы)

Когда ядро работает в автоматическом режиме (после успешной верификации в контуре YMS-MMM), локальная модель выполняет следующую последовательность:

```
ВЕРИФИЦИРОВАННОЕ ЗНАНИЕ получено
        │
        ▼
ОПРЕДЕЛИТЬ МАКРО-КОРЕНЬ
  Вопрос: "К какой из 32 областей это относится первично?"
  Если к одной → символ
  Если к нескольким + confirmed_by >= 3 → лигатура
        │
        ▼
ОПРЕДЕЛИТЬ НАДСТРОЧНЫЙ И ПОДСТРОЧНЫЙ
  Спускаться по фрактальному дереву
  Если уровня не существует → создать, зафиксировать
        │
        ▼
ПОИСК ДУБЛЕЙ В PGVECTOR
  similarity > 0.95 → обновить существующий (Тип А)
  similarity 0.75-0.95 → новый с evolved_from
  similarity < 0.75 → новый независимый
        │
        ▼
ПРИСВОИТЬ НОМЕР (max существующих + 1)
        │
        ▼
ЗАПОЛНИТЬ МЕТАДАННЫЕ
  Все поля обязательны. Пустых полей нет.
        │
        ▼
СЖАТЬ ZSTD + ЗАГРУЗИТЬ НА ШАРД
  Асинхронно. Пользователь не ждёт.
        │
        ▼
ЗАПИСАТЬ ВЕКТОР В PGVECTOR
  Вектор = эмбеддинг от (label + science + section + subsection)
        │
        ▼
ЗАФИКСИРОВАТЬ ПЕРЕХОДНЫЙ МАЯК
  Если evolved_from существует:
  evolution_note = "было [старый метод] → стало [новый метод]"
        │
        ▼
ЗАПУСТИТЬ ХУК-ДОПРОС ФЛАГМАНА
  "...или есть что-то ещё новее?"
```

## Принцип седьмой: Правила которые нельзя нарушать

1. **Рейтинг только инкрементируется.** Никогда не сбрасывается. Перезапись ячейки = +1 к рейтингу.

2. **Базовые 32 символа неизменяемы.** Новые корни не создаются. Всё новое знание помещается внутрь существующих.

3. **Лигатура создаётся только при confirmed_by >= 3.** Если меньше — это символ, не лигатура.

4. **Тело хранится в оригинале.** Никакого нейросетевого квантования тела знания. Только zstd. Байт-в-байт восстановление обязательно.

5. **Хронология не удаляется.** `evolved_from`, `evolution_note`, `last_updated` — неприкосновенны. Удаление или перезапись этих полей = Критическая Ошибка YMS-MMM.

6. **Гиперлинки вшиваются в тело.** Связанные знания адресуются прямо внутри `zstd`-текста в формате `[[EVO:адрес]]`. Это позволяет декомпрессору параллельно подтянуть связанные ячейки без повторного обращения к библиотеке.

7. **Дублей не создавать.** Перед записью обязательна проверка similarity. Мир уже имеет достаточно копий одного и того же.

## Пример полного цикла: от знания до записи

**Входное знание:** Алгоритм обхода капчи в ZennoPoster для Honkai Star Rail с проверкой через n8n webhook.

**Шаг 1 — Корень:** Первично это технология → `τ`

**Шаг 2 — Надстрочный:** Это автоматизация → `^auto`

**Шаг 3 — Подстрочный:** Среда — ZennoPoster → `_zp`

**Шаг 4 — Номер:** В подразделе `τ^auto_zp` уже есть 0046 → присваиваем `0047`

**Шаг 5 — Метаданные:**
```json
{
  "id": "τ^auto_zp_0047",
  "label": "ZennoPoster обход капчи HSR + верификация через n8n webhook",
  "science": "Технология",
  "section": "Автоматизация",
  "subsection": "ZennoPoster",
  "rating_frequency": 1,
  "confirmed_by": 1,
  "evolved_from": "τ^auto_zp_0031",
  "evolution_note": "Переход с Genshin → HSR; добавлена n8n верификация",
  "hyperlinks": ["τ^auto_n8n_0033"],
  "shard_link": {
    "host": "shard-01.free-cloud.io",
    "path": "/evo/TAU/auto/zp/0047.zst",
    "mirror": "shard-01-mirror.free-cloud.io"
  }
}
```

**Шаг 6 — Тело на шарде (содержимое 0047.zst до сжатия):**
```
# ZennoPoster — Обход капчи HSR + n8n верификация

## Условия применения
Платформа: Honkai Star Rail аккаунт-парсинг
Версия ZP: 7.x+
Зависимости: [[EVO:τ^auto_n8n_0033]] — n8n webhook триггер

## Алгоритм
1. Инициализировать сессию браузера с профилем...
[точная инструкция]

## Верификация
После выполнения отправить POST на [[EVO:τ^auto_n8n_0033:step_verify]]
```

**Шаг 7 — Проверка similarity:** Ближайший — `τ^auto_zp_0031` (0.81). Создаём новый с `evolved_from`.

**Результат:** Символ `τ^auto_zp_0047` записан в pgvector. Тело сжато и на шарде.
