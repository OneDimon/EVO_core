## Схемы данных

```python
from pydantic import BaseModel
from typing import Optional

class VerifyRequest(BaseModel):
    session_id: str
    output: str                  # финальный вывод флагмана
    original_tz: str             # исходное ТЗ пользователя
    cartridge: dict              # картридж выданный флагману
    applied_stack: list[str]     # стек применённый флагманом
    solution_quality: str        # "ideal" | "adapted" | "gap_filled"
    deviations: Optional[str]    # что изменено (при adapted/gap_filled)
    workability_confirmed: bool  # подтверждение 200 OK / зелёных тестов
    workability_proof: str       # "HTTP 200 OK" / "tests passed" / etc

class VerificationResult(BaseModel):
    session_id: str
    passed: bool
    score: float                 # 0.0–1.0
    action: str                  # "record_confirmation" | "analyze_delta" |
                                 # "record_new_knowledge" | "fix" | "reanimate"
    failures: list[str]          # конкретные нарушения если есть
    delta_type: Optional[str]    # "A" | "B" — тип архивации при delta
    fix_directive: Optional[str] # что исправить при провале
```

## Алгоритм верификации (полная последовательность)

### Шаг 1 — Запрос подтверждения работоспособности

Сразу при получении отчёта флагмана — до любых проверок:

```
→ Выдать флагману:
  "Проверь работоспособность. Не код — реальный запуск.
   Нужен: HTTP 200 OK / зелёные тесты / успешный старт сервиса."

→ Ждать workability_confirmed: true
→ Если false или отсутствует — считать провалом, счётчик +1
```

### Шаг 2 — Чеклист YMS-MMM (все пункты обязательны)

```
□ workability_confirmed = true (физическая проверка пройдена)
□ 100% соответствие исходному ТЗ пользователя
□ Выданные инструкции картриджа применены корректно
  (при deviations — зафиксировать что изменено и почему)
□ Синтаксическая корректность кода/конфига/ответа
□ Артефакты генерации отсутствуют:
  символы «о», «н», «г», «ч» в медиа-модулях проверены
□ Хронологические данные целостны
  (даты, версии, порядок событий не нарушены)
□ Версии API и инструментов соответствуют картриджу
  (нет подмены: Gemini 1.5 вместо 2.5, Python 3.10 вместо 3.12 и т.д.)
□ applied_stack соответствует detected_stack сессии
```

### Шаг 3 — Определение action по solution_quality

#### solution_quality = "ideal"
Всё применено как выдано. Работает.

```json
{
  "passed": true,
  "action": "record_confirmation",
  "score": 1.0
}
```

→ Передать результат пользователю
→ R_f всех символов набора += 1
→ applicable_stacks[] дополнить applied_stack
→ Контур Obsidian: фиксировать подтверждение
→ Хук-допрос флагмана

#### solution_quality = "adapted"
Флагман адаптировал решение. Работает.

```json
{
  "passed": true,
  "action": "analyze_delta",
  "delta_type": "A",
  "score": 0.85
}
```

→ Передать результат пользователю
→ Контур Obsidian: анализ дельты → Тип А или Тип Б
→ Хук-допрос флагмана

**Определение типа дельты:**
```
Сравнить вектор нового решения с исходным символом:
  similarity > 0.95 → Тип А (перезапись, улучшение того же)
  similarity 0.75-0.95 → Тип Б (новые обстоятельства, новый символ)
  similarity < 0.75 → Тип Б (принципиально иное решение)
```

#### solution_quality = "gap_filled"
Флагман нашёл решение которого не было в базе.

```json
{
  "passed": true,
  "action": "record_new_knowledge",
  "score": 0.9
}
```

→ Передать результат пользователю
→ Контур Obsidian: записать новый символ с нуля
→ Хук-допрос флагмана

#### Провал (1–2 раза)

```json
{
  "passed": false,
  "action": "fix",
  "score": 0.3,
  "failures": ["workability_confirmed = false", "version mismatch: expected 2.5 got 1.5"],
  "fix_directive": "Повтори запуск и убедись в 200 OK. Версия Gemini должна быть 2.5."
}
```

→ Вернуть флагману с fix_directive
→ Счётчик провалов += 1

#### 3 провала подряд — реаниматор

```json
{
  "passed": false,
  "action": "reanimate",
  "score": 0.0
}
```

→ Webhook в [БЛОК 07](BLOCK_07_immune_system.md):
```json
{
  "taskDescription": "<ТЗ>",
  "baseInstructions": "<картридж>",
  "faultyOutput": "<вывод флагмана>",
  "errorLog": "<список failures>",
  "callbackUrl": "/api/v1/patch_callback"
}
```

## Контур Obsidian (архивация после верификации)

Контур Obsidian — зона логики и связей. Запускается после каждого PASS.
Работает асинхронно. Пользователь уже получил ответ.

```
КОНТУР OBSIDIAN:

1. АНАЛИЗ: «Почему это сработало?»
   Сформулировать когнитивную логику решения
   Определить место в фрактальном дереве (наука → раздел → подраздел)

2. КЛАССИФИКАЦИЯ (Тип А или Тип Б):
   ТИП А — улучшение существующего:
     - Новое тело → zstd на лету → шард (асинхронно)
     - ID старого → legacy_symbols[] основного символа
     - R_f += 1
     - evolution_note: "было [X] → стало [Y]"
     - Обновить вектор в pgvector

   ТИП Б — новые обстоятельства/инструменты:
     - Новый символ с evolved_from = ID родительского
     - applicable_stacks[] = applied_stack задачи
     - В теле: [[EVO:родительский_id]]
     - Новый вектор в pgvector

   НОВЫЙ (gap_filled):
     - Определить макро-корень из 32
     - Определить надстрочный и подстрочный
     - Номер: max в подразделе + 1
     - Все метаданные заполнить
     - applicable_stacks[] = applied_stack

3. ПРОВЕРКА НА ЛИГАТУРУ:
   Если confirmed_by символа только что достигло 3:
   → Создать лигатуру из confirmed_in
   → hyperlinks[] = все исходные символы
   → R_f лигатуры = сумма R_f составных
   → Записать в pgvector

4. ПЕРЕХОДНЫЙ МАЯК:
   evolution_note обязателен если есть evolved_from

5. СТАТИСТИКА ГРАФА:
   Пересчитать вес узла (R_f)
   Добавить новые связи
   Обновить applicable_stacks[] символа

6. ХУК-ДОПРОС:
   Выдать флагману:
   "...или есть что-то ещё новее по этой теме применимое к стеку [applied_stack]?"
   
   При has_update: true → повторить архивацию с новым знанием
   При migration_scope: partial/full → подготовить план миграции для пользователя
```

## Правила YMS-MMM (нарушение = Критическая Ошибка)

1. Физическая проверка работоспособности обязательна — до любых других проверок
2. 100% соответствие ТЗ — не «примерно», не «в целом»
3. Подмена версий = критическая ошибка
4. Хронология неприкосновенна
5. Артефакты генерации недопустимы
6. Счётчик провалов — на уровне сессии, не сбрасывается между шагами
7. При 3 провалах — только реаниматор, никаких дополнительных попыток самостоятельно
