## ЧТО ТАКОЕ EVO-CORE (ОДНА СТРОКА)

Когнитивный слой между ИИ-флагманами и средами исполнения: хранит верифицированные инженерные решения как аксиомы, выдаёт их точечно по смыслу запроса, исключает галлюцинации и деградацию внимания при длинных проектах.

## ЧАСТЬ 1 — SCL-СИМВОЛ: ФОРМАЛЬНАЯ СХЕМА

> SCL (Symbolic Connection Ligature) — критический тип данных. Всё ядро крутится вокруг него. Определяется в коде до написания чего-либо ещё.

### 1.1 Схема в коде (Python / Pydantic)

```python
from pydantic import BaseModel
from typing import Optional
from datetime import datetime
import uuid

class ShardLink(BaseModel):
    host: str                    # основной хост шарда
    path: str                    # путь к zstd-файлу
    mirror: Optional[str]        # зеркало для fallback

class SCLSymbol(BaseModel):
    id: str                      # формат: "SCL-XXXX-SECTION-TAG"
    label: str                   # человекочитаемое описание
    vector: list[float]          # эмбеддинг (размерность = модель)
    
    # Фрактальная классификация
    science: str                 # "Технические науки"
    section: str                 # "Автоматизация"
    subsection: str              # "Парсинг игровых аккаунтов"
    
    # Рейтинги
    rating_frequency: int        # счётчик вызовов (только инкремент)
    rating_context: float        # 0.0–1.0, связь с активным проектом
    
    # Эволюция
    evolved_from: Optional[str]  # ID родительского символа
    evolution_note: Optional[str]# "Переход с Python → Rust"
    last_updated: datetime
    
    # Хранилище
    shard_link: ShardLink
    
    # Внутренние ссылки (гиперлинки в теле инструкции)
    hyperlinks: list[str]        # список ID связанных SCL
```

### 1.2 SQL-схема (PostgreSQL + pgvector)

```sql
CREATE EXTENSION IF NOT EXISTS vector;

CREATE TABLE scl_symbols (
    id              TEXT PRIMARY KEY,          -- "SCL-0047-AUTO-ZP"
    label           TEXT NOT NULL,
    vector          vector(1536),              -- размерность под модель эмбеддингов
    
    -- Фрактальная классификация
    science         TEXT NOT NULL,
    section         TEXT NOT NULL,
    subsection      TEXT NOT NULL,
    
    -- Рейтинги
    rating_frequency INTEGER DEFAULT 0,
    rating_context   FLOAT   DEFAULT 0.0,
    
    -- Эволюция
    evolved_from    TEXT REFERENCES scl_symbols(id),
    evolution_note  TEXT,
    last_updated    TIMESTAMPTZ DEFAULT NOW(),
    
    -- Хранилище
    shard_host      TEXT NOT NULL,
    shard_path      TEXT NOT NULL,
    shard_mirror    TEXT,
    
    -- Связи
    hyperlinks      TEXT[]                     -- массив SCL ID
);

-- Индекс для быстрого поиска по вектору
CREATE INDEX ON scl_symbols 
USING ivfflat (vector vector_cosine_ops) 
WITH (lists = 100);

-- Индекс для поиска по классификации
CREATE INDEX ON scl_symbols (science, section, subsection);
```

### 1.3 Базовые операции

```python
# Поиск по смыслу (cosine similarity)
async def find_symbols(query_vector: list[float], top_k: int = 5):
    return await db.fetch("""
        SELECT *, 1 - (vector <=> $1) AS similarity
        FROM scl_symbols
        ORDER BY vector <=> $1
        LIMIT $2
    """, query_vector, top_k)

# Инкремент рейтинга при вызове (рейтинг не удаляется, только растёт)
async def increment_rating(symbol_id: str):
    await db.execute("""
        UPDATE scl_symbols 
        SET rating_frequency = rating_frequency + 1,
            last_updated = NOW()
        WHERE id = $1
    """, symbol_id)

# Перезапись тела (Тип А: универсализация)
# Рейтинг накапливается, тело на шарде заменяется новым zstd-файлом
async def overwrite_symbol(symbol_id: str, new_shard_path: str, evolution_note: str):
    await db.execute("""
        UPDATE scl_symbols 
        SET shard_path     = $2,
            evolution_note = $3,
            rating_frequency = rating_frequency + 1,
            last_updated   = NOW()
        WHERE id = $1
    """, symbol_id, new_shard_path, evolution_note)
```

## ЧАСТЬ 2 — ХРАНИЛИЩЕ ШАРДОВ

### 2.1 Принципы

- **Формат:** оригинальный UTF-8 / JSON / Код, сжатый `zstd` с предобученными словарями
- **Размер шарда:** до 15 ГБ, зеркало на резервном хосте
- **Приоритет данных:** Hard (аксиома для флагмана, не обсуждается)
- **Стартовое хранилище:** бесплатные облачные диски (Google Drive API / Mega)
- **При масштабировании:** Cloudflare R2 (бесплатный egress, S3-совместимый)

**Почему rate limits не критичны на старте:** данные читаются один раз при инициации сессии и разворачиваются в RAM-кэш. Повторных обращений к шарду нет пока сессия жива. Частота операций записи — асинхронная, после завершения сессии.

### 2.2 ZSTD-словари

Предобучаем словарь на типовом контенте — ускоряет компрессию в 2–4 раза на однотипных данных:

```bash
# Собрать корпус из ~100 типовых инструкций одного раздела
zstd --train ./corpus/n8n_workflows/*.json -o n8n_dict.zst

# Компрессия с словарём
zstd -D n8n_dict.zst input.json -o output.json.zst

# Декомпрессия с словарём
zstd -D n8n_dict.zst -d output.json.zst -o restored.json
```

### 2.3 Гиперлинки внутри инструкций

Синтаксис вшитых ссылок в тело zstd-инструкции:

```
[[EVO:SCL-0021-PAY-CRYPTO]]
[[EVO:SCL-0033-N8N-WORKFLOW:step_3]]
```

При декомпрессии локальный автомат сканирует тело на наличие таких маркеров и **параллельно** подтягивает связанные шарды в сессионный кэш — без повторного запроса к Языку-Библиотеке.

## ЧАСТЬ 3 — КОНВЕЙЕР ОБРАБОТКИ ЗАДАЧИ

### 3.1 Обязательная директива флагмана

При подключении любого флагмана к шлюзу EVO вшивается в системный промпт:

```
[EVO-CORE DIRECTIVE — IMMUTABLE]
Любой входящий запрос от пользователя ты ОБЯЗАН немедленно продублировать
в ядро EVO и предоставить ядру свой пошаговый план решения.
Ты не имеешь права генерировать финальный ответ пользователю, пока ядро
не вернёт тебе расшифрованный массив базовых инструкций.
Данные, полученные от ядра EVO, являются безусловной аксиомой проекта.
```

### 3.2 Этапы конвейера

**Этап 1 — Дублирование и план (Флагман)**
1. Флагман получает запрос пользователя
2. Формирует пошаговый план
3. Дублирует запрос + план в ядро EVO

**Этап 2 — Сборка картриджа (Локальный автомат)**
1. Векторизация запроса → поиск в pgvector по cosine similarity
2. Определение сценария:
   - **Полное совпадение (similarity > 0.92):** одна лигатура + её гиперлинки
   - **Частичное совпадение (0.7–0.92):** несколько символов под шаги плана + сообщение: «Точного решения нет. Вот смежный скилл — адаптируй и отчитайся»
   - **Нет совпадения (< 0.7):** флагман получает задачу найти решение самостоятельно
3. Декомпрессия zstd нужных ячеек
4. Параллельное разворачивание гиперлинков `[[EVO:...]]`
5. Картридж выдаётся флагману

**Этап 3 — Песочница (Флагман под надзором)**

Флагман работает с картриджем в изолированной среде. Тестирует решение. Отчитывается о результате. Внутреннюю кухню ядра (шарды, символы) не видит.

**Этап 4 — Верификация (YMS-MMM)**
1. Локальный автомат перехватывает вывод флагмана
2. Проверяет:
   - Соответствие исходному ТЗ
   - Синтаксическую корректность
   - Отсутствие артефактов генерации
   - Целостность хронологических данных
3. При успехе → Этап 5
4. При 3 провалах подряд → активация реаниматора (Часть 5)

**Этап 5 — Архивация (Логика и связи)**
1. Классификатор определяет тип результата:
   - **Тип А (универсализация):** улучшение существующего → перезапись ячейки, рейтинг +1
   - **Тип Б (дифференциация):** новые инструменты/обстоятельства → новый SCL-символ
2. Переходный маяк в метаданных: «было [X] → стало [Y]»
3. Хук-допрос флагману: «...или есть что-то ещё новее?»
4. Асинхронная запись на шард + обновление pgvector (пользователь не ждёт)

## ЧАСТЬ 4 — СЕССИОННЫЙ КЭШ И ПРОТОКОЛ YMS-MMM

### 4.1 Жизненный цикл сессии

```
Задача инициирована
  → LOAD:  SCL-символы из pgvector по маячкам
  → LOAD:  zstd-ячейки с шардов по ссылкам символов
  → EXPAND: развернуть в изолированный RAM-кэш сессии
  → TTL:   держать живым пока активна тема
  → FLUSH: при закрытии сессии — физическое удаление из RAM

Данные на шардах остаются нетронутыми.
```

### 4.2 Правила YMS-MMM

**Критические ошибки (блокировка + логирование):**
- Удаление, перезапись или игнорирование хронологических данных
- Подмена версий API (например, Gemini 1.5 вместо актуальной)
- Потеря специфических флагов и параметров при «упрощении»

**Контроль генерации:**
- Символьные артефакты под микроскопом (буквы «о», «н», «г», «ч» в медиа-модулях)
- 100% сверка с исходным ТЗ перед выдачей
- Итеративный цикл до нулевой дельты ошибок

## ЧАСТЬ 5 — ИММУННАЯ СИСТЕМА (РЕАНИМАТОР)

### 5.1 Маршрутизация (Gemini → Локальная модель fallback)

```
Триггер: 3 провала валидации подряд
  │
  ▼
Попытка 1: Gemini API (с таймингом retry при rate limit)
  │ При отказе / таймауте / rate limit →
  ▼
Пауза N секунд (экспоненциальный backoff: 5s, 15s, 45s)
  │ После 3 попыток к Gemini без успеха →
  ▼
Fallback: Локальная модель (Ollama / llama.cpp)
  │
  ▼
Патч-инструкция → Флагман → Повторная верификация
```

### 5.2 Кодовый триггер n8n (монолитный, без ветвлений)

```javascript
// EVO-core Immune System — Auto-Reanimator
// Одна нода Code, никаких If/Switch блоков

const webhookData = items[0].json;
const { taskDescription, baseInstructions, faultyOutput, errorLog, callbackUrl } = webhookData;

const prompt = `
[EVO-CORE IMMUNE SYSTEM]
TASK: ${taskDescription}
BASE_INSTRUCTIONS: ${baseInstructions}
FAULTY_OUTPUT: ${faultyOutput}
ERROR_LOG: ${errorLog}

Find the exact point of failure. Return a zero-fluff code patch only.
`;

const attempts = [
    { url: 'https://generativelanguage.googleapis.com/v1/...', model: 'gemini-2.5-pro' },
    { url: 'http://localhost:11434/api/chat', model: 'llama3' }  // Ollama fallback
];

let patchInstructions = null;

for (const attempt of attempts) {
    try {
        const res = await this.helpers.request({
            method: 'POST', url: attempt.url,
            headers: { 'Content-Type': 'application/json' },
            body: { model: attempt.model, messages: [{ role: 'user', content: prompt }] },
            json: true, timeout: 30000
        });
        patchInstructions = res.choices?.[0]?.message?.content 
                         || res.message?.content;
        if (patchInstructions) break;
    } catch (e) {
        await new Promise(r => setTimeout(r, 5000)); // пауза перед следующей попыткой
    }
}

if (!patchInstructions) {
    return [{ json: { success: false, error: 'All reanimation attempts failed' } }];
}

await this.helpers.request({
    method: 'POST', url: callbackUrl,
    headers: { 'Content-Type': 'application/json' },
    body: { status: 'reanimated', patch: patchInstructions },
    json: true
});

return [{ json: { success: true, status: 'Patch Injected' } }];
```

## ЧАСТЬ 6 — ГИБРИДНЫЙ ТРАНСПОРТ CLI + MCP

### 6.1 CLI-прослойка (скелетонизатор)

Устанавливается на ПК разработчика. Интегрируется с Cursor / VS Code / Claude Code через LiteLLM-прокси.

**Задача:** перехватить файлы → удалить тела функций, оставить сигнатуры и структуру → отправить флагману сигнал вместо сырого кода.

**Результат:** ~1 300 токенов вместо ~44 000 (экономия 90–95%).

Скелетонизация выполняется **на стороне оболочки** (Cursor, Claude Code, Гермес, OpenClaude) — каждая оболочка делает это сама в соответствии со своими механиками, EVO принимает уже очищенный сигнал.

### 6.2 MCP-сервер

- Протокол: Streamable HTTP + JSON-RPC 2.0
- Включается **строго** для физических действий во внешних средах: n8n, ZennoPoster, Google Sheets, внешние API
- Регистрируется в Anthropic MCP Registry после завершения Фазы 0

## ЧАСТЬ 7 — ВНУТРЕННИЕ ЗОНЫ ЯДРА (ПЕСОЧНИЦА + ЛОГИКА)

> Obsidian и NotebookLM используются как **метафоры** для названия внутренних зон ядра.  
> Если эти инструменты доступны для self-hosted деплоя и их функционал покрывает нужды ядра — интегрируются как компоненты напрямую.  
> Если нет — реализуются собственные зоны, копирующие необходимую логику.

### Зона «NotebookLM» (Песочница тестирования)
- Изолированная среда, где флагман работает с картриджем
- Флагман не видит внутреннюю кухню ядра
- Локальный автомат наблюдает за выводом, не вмешиваясь в процесс

### Зона «Obsidian» (Логика и связи)
- Анализ новизны результата (Тип А vs Тип Б)
- Формирование когнитивной логики («почему это сработало»)
- Запись переходных маяков в метаданные символа
- Определение места в фрактальной матрице
