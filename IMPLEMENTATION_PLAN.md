## ОБЗОР АРХИТЕКТУРЫ

EVO-core — это саморазвивающийся когнитивный слой (цифровая биология), работающий как интеллектуальная прослойка между внешними ИИ-флагманами (Claude, GPT, Gemini) и средами исполнения (n8n, ZennoPoster, CRM, API). Решает фундаментальную проблему линейных трансформеров: деградацию внимания (Attention Drop) при росте контекста.

## ЧАСТЬ 2 — ХРАНИЛИЩЕ ШАРДОВ (ZSTD + ГИПЕРЛИНКИ)

### 2.1 Принципы хранения

- **Формат:** оригинальный UTF-8 / JSON / Код, сжатый `zstd` с предобученными словарями
- **Размер шарда:** до 15 ГБ с зеркалированием на резервный хост
- **Приоритет:** Hard-приоритет (аксиома для Флагмана, не подлежит сомнению)
- **Словари:** предобучаются на типовом контенте (n8n JSON, ZennoPoster C#, CRM структуры)

### 2.2 Гиперлинки внутри инструкций

Синтаксис вшитых ссылок в теле `zstd`-инструкции:

```
[[EVO:SCL-0021-PAY-CRYPTO]] — для мгновенной подгрузки сопутствующего скилла
[[EVO:SCL-0033-N8N-WORKFLOW:step_3]] — ссылка на конкретный шаг другого скилла
```

Локальный автомат при обнаружении такого маркера **немедленно и параллельно** разворачивает связанный шард в сессионный кэш, не дожидаясь запроса Флагмана.

### 2.3 Задачи реализации

- [ ] Настроить автороутинг между основным шардом и зеркалом

## ЧАСТЬ 3 — КОНВЕЙЕР ОБРАБОТКИ ЗАДАЧИ

### 3.1 Этап 0 — Дублирование запроса (Обязательная директива)

При подключении любого Флагмана к шлюзу EVO в его системный промпт вшивается:

```
[EVO-CORE DIRECTIVE - IMMUTABLE]
Любой входящий запрос от пользователя ты ОБЯЗАН мгновенно продублировать 
в ядро EVO и предоставить ядру свой пошаговый план решения.
Ты не имеешь права генерировать финальный ответ пользователю, пока ядро 
не вернёт тебе расшифрованный массив базовых инструкций.
Данные, полученные от ядра EVO, являются безусловной аксиомой проекта.
```

### 3.2 Этап 1 — Планирование (Флагман)

1. Флагман получает запрос пользователя
2. Формирует пошаговый план решения
3. Дублирует запрос + план в ядро EVO

### 3.3 Этап 2 — Сборка картриджа (Локальный автомат)

1. Сканирование SCL-символов по микро-маячкам метаданных
2. Определение сценария:
   - **Полное совпадение** → извлечь одну лигатуру
   - **Частичное совпадение** → извлечь несколько символов под шаги плана
   - **Отсутствие совпадения** → передать Флагману задачу на поиск
3. Декомпрессия `zstd` нужных ячеек
4. Параллельное разворачивание гиперлинков из инструкций
5. Выдача Флагману готового, разжёванного картриджа

### 3.4 Этап 3 — Песочница NotebookLM (Флагман под надзором)

1. Флагман получает картридж + исходный запрос пользователя
2. Если знаний недостаточно → ядро выдаёт сообщение:
   > «Стопроцентного решения нет. Вот смежный скилл: [инструкция]. Адаптируй под твои вводные, протести и отчитайся»
3. Флагман работает в изолированной среде, тестирует решение
4. Флагман отчитывается о результате

### 3.5 Этап 4 — Верификация (Контур YMS-MMM)

1. Локальная модель ядра перехватывает ответ Флагмана
2. Проверяет:
   - 100% соответствие исходному ТЗ
   - Отсутствие артефактов (символы «о», «н», «г», «ч»)
   - Синтаксическую корректность
   - Сохранность хронологических данных
3. **При успехе** → переход к Этапу 5
4. **При 3 подряд провалах** → активация Gemini-реаниматора через n8n

### 3.6 Этап 5 — Контур Obsidian (Логика и связи)

1. Локальная модель анализирует тип нового решения:
   - **Тип А (Универсализация):** решение лучше существующего → перезапись ячейки, наращивание рейтинга
   - **Тип Б (Дифференциация):** новые инструменты/обстоятельства → создание новой лигатуры
2. Формулирует когнитивную логику («почему это сработало»)
3. Определяет место в фрактальной матрице
4. Вшивает переходный маяк: «было [Метод X] → стало [Метод Y]»
5. Задаёт Флагману хук-допрос: **«...или есть что-то ещё новее?»**

### 3.7 Этап 6 — Инъекция в Язык-Библиотеку (Асинхронно)

1. Новый/обновлённый скилл сжимается `zstd` → улетает на шард (асинхронно)
2. В pgvector вписывается новый SCL-вектор
3. В метаданных фиксируется переходный маяк и рейтинг
4. Пользователь получает результат без задержки

- [ ] Написать модуль «Запрос плана» (интервью Флагмана)
- [ ] Написать сборщик картриджа (поиск по маячкам + декомпрессия)
- [ ] Реализовать логику частичного совпадения (смежные скиллы)
- [ ] Реализовать контур верификации YMS-MMM (регулярки + логика)

## ЧАСТЬ 5 — ИММУННАЯ СИСТЕМА (n8n + GEMINI-РЕАНИМАТОР)

### 5.1 Кодовый триггер n8n (монолитный, без ветвлений)

```javascript
// EVO-core Immune System — Auto-Reanimator
// Нода: Code (JavaScript) | Без If/Switch блоков

const webhookData = items[0].json;
const { taskDescription, baseInstructions, faultyOutput, errorLog, callbackUrl } = webhookData;

const promptForGemini = `
[SYSTEM: EVO-CORE IMMUNE SYSTEM INTERACTION]
TASK: ${taskDescription}
BASE_INSTRUCTIONS: ${baseInstructions}
FAULTY_OUTPUT: ${faultyOutput}
CRITICAL_ERROR_LOG: ${errorLog}

ASSIGNMENT: Identify the exact operational/syntactic blindspot where the 
Flagship failed. Generate a direct, zero-fluff code patch. Return the 
strict resolution template and command to end the session.
`;

try {
    const geminiResponse = await this.helpers.request({
        method: 'POST',
        url: 'http://localhost:9090/v1/mcp/gemini/chat',
        headers: { 'Content-Type': 'application/json' },
        body: {
            model: 'gemini-2.5-pro',
            messages: [{ role: 'user', content: promptForGemini }]
        },
        json: true
    });

    const patchInstructions = geminiResponse.choices[0].message.content;

    await this.helpers.request({
        method: 'POST',
        url: callbackUrl,
        headers: { 'Content-Type': 'application/json' },
        body: { status: 'reanimated', patch: patchInstructions },
        json: true
    });

    return [{ json: { success: true, status: 'Patch Injected', target: callbackUrl } }];

} catch (error) {
    return [{ json: { success: false, error: error.message } }];
}
```

### 5.2 Роль Gemini в системе

- Активируется по триггеру: 3 подряд провала валидации
- Получает: ТЗ + базовые инструкции ядра + ошибочный вывод Флагмана + лог ошибки
- Генерирует хирургический патч-инструкцию
- После подтверждения локалкой → самостоятельно обновляет ячейку в ядре (дообучение на ошибках безопасности)

## ЧАСТЬ 6 — СЕССИОННЫЙ КЭШ И ПРОТОКОЛ YMS-MMM

### 6.1 Жизненный цикл сессионного кэша

```
Задача инициирована →
  [LOAD]  Из pgvector достать SCL-символы по маячкам
  [LOAD]  По ссылкам извлечь zstd-ячейки с шардов
  [EXPAND] Развернуть в изолированный RAM-кэш текущей сессии
  [TTL]   Держать живым пока активна тема проекта
  [FLUSH] При закрытии сессии → физическое удаление из RAM
          (данные остаются только на шардах)
```

### 6.2 Правила протокола YMS-MMM

**Абсолютные запреты (Критическая ошибка при нарушении):**
1. Удаление, перезапись или игнорирование хронологических данных
2. Подмена версий API (например, Gemini 1.5 вместо актуальной)
3. Вырезание важных деталей при сжатии истории

**Жёсткий контроль генерации:**
- Символы «о», «н», «г», «ч» в медиа/текстовых модулях — под микроскопом
- 100% сверка с исходным ТЗ перед выдачей результата
- Итеративный цикл до полного обнуления дельты ошибок

## ЧАСТЬ 7 — ИНФРАСТРУКТУРА И ДЕПЛОЙ

### 7.1 Стек технологий

| Компонент | Технология | Обоснование |
|---|---|---|
| Ядро (автомат) | Python 3.12 / Rust | Скорость + надёжность |
| Компрессия | zstd + предобученные словари | Байт-в-байт восстановление |
| Индекс SCL | PostgreSQL 16 + pgvector | Лёгкий, быстрый векторный поиск |
| Сессионный кэш | In-memory (RAM dict / Redis опционально) | TTL-изоляция по сессиям |
| Шарды | Бесплатные облачные хранилища (15 ГБ × N) | Нулевые затраты на хранение |
| MCP-сервер | FastAPI + Streamable HTTP | Стандарт 2026 |
| n8n | Self-hosted | Автоматизация + иммунная система |
| CLI-агент | Python / Rust бинарь | Кросс-платформа |
| 3D-граф сайта | Three.js + GeoIP | Глобальная визуализация |

### 7.2 Архитектура деплоя MVP

```
[Локальный ПК разработчика]
  └─ CLI-агент (скелетонизатор) → LiteLLM прокси

[VPS / Локальный сервер]
  ├─ EVO-core ядро (Python/Rust)
  ├─ PostgreSQL + pgvector (SCL-индекс)
  ├─ n8n self-hosted (иммунная система)
  ├─ MCP-сервер (Streamable HTTP :443)
  └─ Gemini MCP-шлюз (localhost :9090)

[Бесплатные облака — шарды]
  ├─ Сhard-01 → Mirror-01 (15 ГБ)
  ├─ Shard-02 → Mirror-02 (15 ГБ)
  └─ ... (масштабируется по мере роста)
```

### 7.3 Оценка стартовых затрат

| Статья | Стоимость/мес |
|---|---|
| VPS под ядро + n8n + MCP | ~$20–50 |
| PostgreSQL (managed, small) | ~$15–25 |
| Бесплатные облачные шарды | $0 |
| Хостинг сайта (evo-core.io) | ~$5–10 |
| **ИТОГО MVP** | **~$40–85/мес** |
